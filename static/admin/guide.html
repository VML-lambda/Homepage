    콘텐츠 수정 방법

    연구실 구성원은 GitHub 계정이 있으면 홈페이지 콘텐츠를 직접 수정할 수 있습니다. 별도 설치 없이 웹 브라우저에서만 작업합니다.

    1관리 페이지 접속 및 로그인

    /admin/ 페이지에 접속한 뒤 Login with GitHub 버튼을 클릭합니다.

    GitHub 계정으로 로그인하면 자동으로 권한이 확인됩니다. repo에 write 권한이 있어야 편집이 가능합니다.

    2수정할 항목 선택

    왼쪽 사이드바에서 카테고리를 선택합니다.

    카테고리내용
    AboutAbout 페이지 텍스트·이미지 수정
    Research연구 팀별 상세 페이지 (NVC, VCM, 3DV)
    Papers논문 목록 (국제/국내 학술지·학술대회)
    Standard ContributionsMPEG·JVET·JPEG 표준 기여 목록
    News연구실 소식 (논문 accept, 수상, 행사 등)
    Students재학생 정보
    Alumni졸업생 정보
    Grants연구과제(지원금) 목록
    3항목 수정 또는 추가

    기존 항목을 클릭하면 편집 화면이 열립니다. 새 항목은 우측 상단 New [항목명] 버튼을 클릭합니다.

    모든 필드를 작성한 뒤 우측 상단 Save를 누릅니다.

    이미지 업로드: Image 필드에서 Choose an image를 클릭하면 미디어 라이브러리가 열립니다. 기존 이미지를 선택하거나 새 파일을 업로드할 수 있습니다.
    4저장 후 배포 요청

    Save하면 변경사항이 GitHub의 cms-draft 브랜치에 저장됩니다. 홈페이지에 바로 반영되지 않습니다.

    CMS에서 수정 후 Publish
    →
    cms-draft 브랜치에 자동 커밋
    →
    관리자에게 배포 요청
    →
    PR 머지 → 자동 배포

    여러 항목을 수정한 뒤 한 번에 관리자에게 배포를 요청하는 것을 권장합니다.

    각 항목 작성 요령

    About 페이지

    About 페이지의 모든 텍스트와 이미지를 CMS에서 수정할 수 있습니다.

    • Research Section Heading: 연구 팀 그리드 제목
    • Introduction Heading / Body: Visual Media 소개 섹션 제목·본문 (마크다운 지원)
    • Checklist Items: 체크 리스트 항목 (추가/삭제/순서 변경 가능)
    • Highlight Box: 왼쪽 파란 바 강조 박스 내용
    • Image 1, 2: 오른쪽 이미지 2장
    연구 팀 카드(NVC, VCM, 3DV)는 Research 컬렉션에서 자동으로 가져옵니다. About 페이지에서 직접 수정하지 않습니다.

    Research (연구 팀 페이지)

    각 연구 팀(NVC, VCM, 3DV) 상세 페이지를 관리합니다. About 페이지의 카드를 클릭하면 이동하는 페이지입니다.

    • Title / Short Name / Subtitle: 팀 이름, 약칭, 한 줄 설명
    • Cover Image: 대표 이미지 (About 카드 + 상세 페이지에 표시)
    • YouTube Video ID: 유튜브 영상 ID (URL의 v= 뒤 부분, 예: dQw4w9WgXcQ)
    • PDF File: 팀 소개 PDF 파일 업로드 (페이지에 임베딩 표시)
    • Overview: 연구 목적·중요성·활용 분야 소개 (마크다운, 영문 권장)
    • Team Lead / Members: Students 목록에서 검색·선택 (드롭다운)
    • Research Highlights: 시간순 연구 성과 목록
      • Title, Period(선택), Description(마크다운), Key Figure(이미지), Key Result, Participants
      • Related Links: 관련 논문·과제·표준 링크 추가 가능
    자동 연결 기능: Papers, Contributions, Grants에 Research Area 태그(NVC/VCM/3DV)를 지정하면, 해당 연구 팀 페이지 하단에 자동으로 표시됩니다. 수동 링크 불필요.
    새 연구 팀을 추가/삭제하는 경우: 상단 메뉴바를 수동으로 수정해야 하므로, 관리자에게 별도로 알려주세요.

    Papers (논문)

    • Year.Month: 2025.03 형식으로 입력 (점 구분)
    • Type: International (국제) / Domestic (국내) 중 선택
    • Co-authors: 쉼표로 구분, 교신저자는 이름 뒤에 <sup>†</sup>로 위첨자 십자 표시
    • Image: 논문 대표 이미지 (없어도 무방)
    • Research Area: NVC / VCM / 3DV 중 선택 (복수 선택 가능, 선택 시 해당 연구 팀 페이지에 자동 표시)
    • Grant: 관련 과제명 (선택)

    Standard Contributions (표준 기여)

    • Organization: 기본값 ISO/IEC JTC 1
    • Org Name: 예) SC29/WG4 (MPEG)
    • Meeting Name: 예) MPEG 149th Meeting
    • Type: MPEG / JVET / JPEG 중 선택
    • Research Area: NVC / VCM / 3DV 중 선택 (복수 선택 가능)
    • Grant: 관련 과제명 (선택)

    News (소식)

    • Category: accept / award / event / notice 중 선택
    • Show on Home: 체크 시 메인 페이지에도 표시됨
    • Body: 마크다운 형식으로 상세 내용 작성 가능

    Students / Alumni (구성원)

    • Display Order: 숫자가 작을수록 먼저 표시
    • Photo: Image 필드에서 Choose an image를 클릭하여 기존 이미지를 선택하거나 새 파일을 업로드합니다.

    배포 파이프라인 구조

    구성원이 CMS 수정
    →
    cms-draft 브랜치에 자동 커밋
    ↓
    PR 자동 생성 GitHub Actions
    →
    리뷰어 자동 지정·알림
    →
    Squash-merge + 브랜치 삭제 관리자 작업
    ↓
    Hugo 빌드 + GitHub Pages 배포 GitHub Actions
    →
    cms-draft 브랜치 자동 재생성 (master 기준)
    자동화 요약: 관리자가 직접 PR을 만들 필요가 없습니다. cms-draft에 푸시가 발생하면 .github/workflows/cms-draft-pr.yml이 PR(cms-draft → master)을 자동 생성하고 리뷰어를 지정합니다. 관리자는 리뷰 후 Squash and merge만 눌러주면 되며, 머지 시 브랜치를 삭제하면 .github/workflows/gh-pages.yml의 sync 단계가 다음 배포 때 cms-draft를 master 기준으로 다시 만들어 둡니다.

    초기 설정 (최초 1회)

    1Cloudflare Workers에 OAuth 프록시 배포
    1. sveltia-cms-auth GitHub에서 "Deploy to Cloudflare Workers" 버튼 클릭
    2. Cloudflare 계정으로 로그인 (무료 계정 가능)
    3. 배포 완료 후 Worker URL 확인 (예: https://sveltia-cms-auth.your-account.workers.dev)
    2GitHub OAuth App 생성
    1. GitHub → Settings → Developer settings → OAuth Apps → New OAuth App
    2. 다음과 같이 입력:
      Application name: VML CMS Homepage URL: https://vmlab.khu.ac.kr Authorization callback URL: https://your-worker.workers.dev/callback
    3. 생성 후 Client ID와 Client Secret 복사
    3Cloudflare Worker 환경변수 설정

    Cloudflare 대시보드 → Workers → sveltia-cms-auth → Settings → Variables에서 다음 값을 설정합니다.

    변수명값
    GITHUB_CLIENT_IDGitHub OAuth App의 Client ID
    GITHUB_CLIENT_SECRETGitHub OAuth App의 Client Secret (Encrypt 권장)
    ALLOWED_DOMAINSvmlab.khu.ac.kr
    4config.yml에 Worker URL 입력

    static/admin/config.yml의 base_url을 Worker URL로 설정합니다.

    backend: name: github repo: VML-lambda/Homepage branch: cms-draft base_url: https://your-worker.workers.dev
    5구성원에게 repo 쓰기 권한 부여

    GitHub → Repository → Settings → Collaborators → Add people
    편집 권한이 필요한 구성원을 Write 권한으로 초대합니다.

    PR 머지 절차 (배포 시)

    1PR 알림 확인 자동

    구성원이 CMS에서 저장하면 cms-draft 브랜치에 자동 커밋되고, .github/workflows/cms-draft-pr.yml이 즉시 실행되어 PR(cms-draft → master)을 자동으로 생성합니다.

    • 제목: merge: CMS updates
    • 라벨: cms-draft
    • 리뷰어: 워크플로우에 지정된 관리자들에게 자동 할당 · 알림 전송

    이미 열린 PR이 있으면 추가로 만들지 않고 기존 PR에 커밋만 쌓입니다. 관리자는 GitHub 알림 또는 Pull requests 탭에서 PR을 확인합니다.

    2내용 검토

    PR의 Files changed 탭에서 변경 내용을 확인합니다. 수정이 필요한 부분이 있으면 구성원에게 CMS에서 다시 편집하도록 요청하거나, 직접 수정 후 cms-draft에 푸시하면 같은 PR에 반영됩니다.

    3Squash and merge + 브랜치 삭제

    이상 없으면 merge 버튼 옆 드롭다운을 열어 Squash and merge를 선택합니다. CMS 편집은 자잘한 자동 커밋이 많기 때문에 squash로 합쳐 master 히스토리를 깔끔하게 유지합니다.

    1. Squash and merge 클릭 → 커밋 메시지 확인 → Confirm squash and merge
    2. 머지 완료 후 나타나는 Delete branch 버튼을 눌러 cms-draft 브랜치를 삭제합니다.
    브랜치를 삭제해도 됩니다. gh-pages.yml의 sync-cms-draft job이 배포 완료 후 cms-draft 브랜치가 없으면 master 기준으로 자동으로 다시 생성합니다. 다음 CMS 편집은 깨끗한 새 cms-draft 위에서 시작됩니다.
    Merge 옵션 주의: Create a merge commit이나 Rebase and merge는 사용하지 마세요. squash merge하여 master 브랜치의 커밋 히스토리를 깔끔하게 유지합니다.
    4배포 확인 자동

    squash-merge 직후 gh-pages.yml이 자동 실행됩니다:

    1. Hugo 빌드
    2. GitHub Pages에 배포 (약 1~2분 소요)
    3. cms-draft 브랜치 동기화 — 기존 브랜치가 있으면 master를 merge, 없으면 master 기준으로 새로 생성

    Actions 탭에서 녹색 체크가 뜨면 https://vmlab.khu.ac.kr에 반영된 것입니다. 실패 시 해당 job 로그를 확인하세요.

    팀원에게 대규모 CMS 수정 요청하기 (임시 미리보기 서버)

    왜 필요한가: CMS에서 저장한 내용은 cms-draft 브랜치에 자동 커밋되지만, 관리자가 PR을 머지해 master에 반영하기 전까지는 라이브 사이트(vmlab.khu.ac.kr)에 나타나지 않습니다. 팀원에게 대규모 내용 수정을 요청할 때는 구성원이 저장 결과를 배포 전에 미리보기로 확인할 수 있어야 하므로, 관리자 PC에서 cms-draft 브랜치를 임시로 호스팅합니다.

    리포지토리 루트의 serve-cms-draft.ps1(Windows) / serve-cms-draft.sh(Linux/macOS) 스크립트가 이 용도로 준비되어 있습니다. Hugo 서버를 0.0.0.0:1313에 바인딩하고, 30초마다 git pull origin cms-draft로 최신 CMS 저장본을 반영합니다.

    절차

    1관리자 PC에서 임시 서버 실행
    1. 관리자 PC를 연구실 Wi-Fi(vmlab5G-1)에 연결합니다.
    2. 리포지토리 루트에서 OS에 맞는 스크립트를 실행합니다.
    # Windows (PowerShell, 리포지토리 루트) powershell -ExecutionPolicy Bypass -File .\serve-cms-draft.ps1 # Linux / macOS chmod +x serve-cms-draft.sh ./serve-cms-draft.sh

    셸을 닫으면 미리보기 서버도 종료됩니다. 요청 기간 동안 셸을 열어두세요.

    2관리자 PC의 내부 IP 확인

    공유할 URL은 http://<관리자PC_IP>:1313/ 형식입니다. vmlab5G-1 네트워크에서 잡힌 IPv4 주소를 확인하세요.

    # Windows ipconfig # "무선 LAN 어댑터 Wi-Fi"의 "IPv4 주소" 확인 (예: 192.168.0.54) # Linux / macOS ip addr # 또는 ifconfig

    공유기·네트워크 구성에 따라 주소는 달라집니다. 위 예시 192.168.0.54는 현재 관리자 PC 기준 값이며, 팀원에게 공유할 최종 URL은 다음과 같은 형태가 됩니다.

    http://192.168.0.54:1313/ # 홈 http://192.168.0.54:1313/about/ # About 페이지 http://192.168.0.54:1313/news/ # News 페이지
    3네트워크 접근 허용 설정 처음 1회

    내부망 다른 기기에서 http://<IP>:1313/이 열리지 않으면 아래 세 가지를 순서대로 확인합니다.

    A. Windows 방화벽 — 인바운드 허용

    • Hugo를 처음 실행할 때 표시되는 Windows Defender 팝업에서 "사설 네트워크"를 체크하고 액세스 허용을 클릭합니다.
    • 이미 거부했다면: 제어판 → Windows Defender 방화벽 → 고급 설정 → 인바운드 규칙 → 새 규칙 → 포트 → TCP 1313 허용 (사설 프로필만 선택 권장)
    • 참고: Microsoft Learn — Windows 방화벽 인바운드 포트 규칙 만들기

    B. 네트워크 프로필을 "사설"로 전환

    • Wi-Fi 프로필이 공용(Public)이면 동일 LAN 기기의 접근이 기본 차단됩니다.
    • Windows: 설정 → 네트워크 및 인터넷 → Wi-Fi → vmlab5G-1 속성 → 네트워크 프로필을 "사설(Private)"로 변경
    • 참고: Microsoft Support — 네트워크 위치(공용/사설) 설정

    C. 공유기의 Client/AP Isolation 비활성화

    • 일부 공유기는 같은 Wi-Fi에 붙은 기기 간 통신을 기본 차단합니다(보안 기능). 이 경우 방화벽 설정과 무관하게 접근 불가.
    • 공유기 관리 페이지의 무선 설정에서 "클라이언트 격리" / "AP Isolation" / "Station Isolation" 항목을 찾아 OFF로 설정합니다.
    • 공유기 관리 권한이 없다면 네트워크 관리자에게 이 항목 변경을 요청하세요.
    • 참고: Wikipedia — Wireless isolation (AP Isolation)
    체크 순서: 같은 네트워크의 다른 기기에서 ping <관리자PC_IP>가 응답하면 공유기 격리는 문제 없음 → 방화벽·포트 문제. ping이 아예 실패하면 공유기 Isolation부터 해제해야 합니다.

    아래 템플릿을 복사해 주제·컬렉션명·경로·IP·마감일만 바꿔 보내세요.

    제약 및 향후 개선

    제약 사항
    · 관리자 PC 전원이 켜져 있고 연구실 Wi-Fi에 연결되어 있어야만 동작
    · 연구실 내부 네트워크에서만 접근 가능 (외부·집에서는 불가)
    · 스크립트가 실행 중인 셸을 닫으면 미리보기 서버도 중단됨
    장기 개선: cms-draft 브랜치를 자동 빌드·호스팅하는 별도 dev 도메인(예: dev.vmlab.khu.ac.kr)을 두는 것이 이상적입니다. 현재는 대규모 수정 요청 빈도가 낮아 임시 서버 방식으로 운영합니다. 외부 검토가 필요하거나 요청이 잦아지면 dev 도메인 구축을 고려하세요.

    자주 있는 상황

    1새 구성원 사진 추가
    1. CMS → Students에서 해당 구성원 편집
    2. Photo 필드에서 Choose an image 클릭 → 이미지 업로드 또는 기존 이미지 선택
    2연구 팀(Research) 추가·삭제 — navbar 수동 업데이트
    CMS Research 컬렉션에 새 팀을 추가하거나 기존 팀을 삭제한 경우, 상세 페이지(/about/{slug}/)와 About 오버뷰 카드는 Hugo가 자동으로 갱신하지만, 상단 네비게이션 바의 About 드롭다운 메뉴는 하드코딩되어 있어 수동으로 layouts/partials/navbar.html을 수정해야 합니다.

    수정 위치: layouts/partials/navbar.html의 About 드롭다운 블록

    <div class="nav-item dropdown"> <a href="#" class="nav-link dropdown-toggle" data-bs-toggle="dropdown">About</a> <div class="dropdown-menu bg-light m-0"> <a href="/about/" class="dropdown-item">Overview</a> <a href="/about/nvc/" class="dropdown-item">NVC</a> <a href="/about/vcm/" class="dropdown-item">VCM</a> <a href="/about/3dv/" class="dropdown-item">3DV</a> <!-- 팀 추가 시: <a href="/about/{slug}/" class="dropdown-item">{ShortName}</a> 줄을 추가 --> </div> </div>
    1. 추가: CMS에서 입력한 slug와 일치하도록 <a href="/about/{slug}/" class="dropdown-item">{표시명}</a> 한 줄을 기존 항목들 아래에 추가
    2. 삭제: 해당 팀의 <a> 줄을 제거
    3. 저장 후 git commit → git push origin master → GitHub Actions가 자동 배포

    CMS 편집만으로는 이 파일이 변경되지 않으므로, 팀원이 Research 컬렉션에 새 팀을 추가한 경우 관리자가 반드시 이 작업을 수행해야 네비게이션에 메뉴가 나타납니다.

    3사이트 레이아웃/디자인 수정

    layouts/ 폴더의 HTML 파일을 직접 수정한 뒤 master에 push합니다. CMS를 거치지 않고 git으로 직접 작업합니다.

    수정 후 반영 방법

    로컬에서 hugo server로 미리보기 확인 후, git commit → master push하면 GitHub Actions가 자동으로 배포합니다. cms-draft 브랜치도 자동 동기화됩니다.
    git add layouts/ static/css/ git commit -m "feat: UI 수정 내용 요약" git push origin master

    Hugo 빌드 에러 발생 시

    로컬에서 hugo --minify를 실행해 에러 메시지를 먼저 확인합니다. 대부분 마크다운 파일의 front matter(---로 감싸진 헤더) 형식 오류입니다.
